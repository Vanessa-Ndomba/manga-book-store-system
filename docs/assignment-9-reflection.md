# Assignment 9: Reflection on Domain Modeling and Class Diagram Development

Designing the domain model and class diagram for MangaBookStore was more challenging than I expected because the system requirements cover both customer-facing e-commerce flows and admin operational workflows. At first, it seemed simple to identify obvious entities such as user, manga, cart, and order. However, the real challenge was deciding what should be represented as a true domain entity versus what should stay as an attribute or service-level behavior. For example, profile and address management are clearly required by FR-15, but adding a dedicated Address class to the first version of the model would increase complexity and distract from the core transaction lifecycle in Assignment 9. I chose to keep address data associated with the user/order context while prioritizing entities that are central to browsing, cart management, checkout, order progression, and stock control.

A second challenge was balancing abstraction and concreteness in method definitions. If methods are too technical (for example, database-oriented CRUD naming), the model stops communicating business meaning. If methods are too high-level (for example, “processEverything”), the model loses design usefulness. I tried to keep methods behavior-focused: `reserveStock`, `checkout`, `capture`, `markShipped`, and `markDelivered` explain what the business does, not how it is coded. This helped align the class diagram with the requirements and with the Assignment 8 state transitions. In particular, payment and order methods were constrained to match realistic transition logic, such as authorization before capture and cancellation boundaries before shipment completion.

The most important alignment task was connecting this Assignment 9 design with previous assignments. From the requirements perspective (Assignment 4 artifacts in this repo), the selected entities map directly to major functional groups: authentication (`UserAccount`), catalog (`MangaListing`), cart (`ShoppingCart`, `CartItem`), order management (`Order`, `OrderItem`), payment (`PaymentTransaction`), and inventory (`InventoryItem`). From use cases (Assignment 5), the model supports registration/login, browsing/searching, adding to cart, checkout, profile updates, and admin maintenance actions. From behavioral modeling (Assignment 8), I intentionally reused lifecycle concepts already documented in state diagrams: cart activation/abandonment/check-out conversion, order progression from created to delivered/cancelled, payment initiation/authorization/capture/refund, and inventory reservation/allocation/replenishment. This gave me confidence that the static structure (class diagram) and dynamic behavior (state/activity diagrams) are coherent rather than contradictory.

I also had to make trade-offs between inheritance and composition. I used inheritance for user roles (`Customer` and `Admin` from `UserAccount`) because they share identity and security properties but differ in responsibilities. That said, role inheritance can become rigid if role combinations or policy changes are needed later. A role/permission composition model (RBAC tables and policies) would be more flexible in a production design, but inheritance communicates intent clearly at this stage and is easier for assignment evaluation. For cart and order line items, composition was the better choice because `CartItem` and `OrderItem` do not make business sense without their parent aggregate. This was a straightforward and clean decision.

Another trade-off involved scope and model simplicity. The repository indicates simulated payment and shipping integration in constraints, so I deliberately avoided over-modeling external systems (gateway adapters, shipment tracking APIs, event buses). Overly detailed technical modeling might look sophisticated, but it can blur domain boundaries and reduce clarity for a coursework deliverable focused on core object-oriented design principles. I also treated `InventoryItem` as tightly bound to `MangaListing` in the class diagram, which simplifies stock reasoning for one-SKU-per-listing behavior. In a larger retail system, inventory may require multi-warehouse modeling and looser coupling.

One major lesson learned is that good object-oriented design is not only about drawing correct UML symbols; it is about preserving consistency across artifacts. Requirements say what must happen, use cases show user interactions, state/activity diagrams describe behavior over time, and class diagrams define structural responsibilities. If these artifacts are produced independently, they can drift and cause contradictions. Building Assignment 9 from existing repository artifacts forced me to think in terms of traceability. Another lesson is that multiplicity is not a cosmetic detail; it captures business constraints directly. Declaring that a customer has many orders or an order has one-to-many order items immediately clarifies rules and prevents ambiguous implementation choices later.

Overall, this assignment improved my confidence in translating product requirements into domain-focused classes while staying aligned with prior analysis and behavior models. I now better understand that object-oriented design quality comes from intentional boundaries, explicit business rules, and disciplined consistency between static and dynamic models, not from diagram size alone.
