# Reflection

## Challenges in Choosing Granularity
One of the biggest challenges in this assignment was deciding how much detail to include in the diagrams. If too many states or actions are shown, the diagrams become difficult to read. However, if too few are included, important logic and business rules may be missed. I solved this by focusing on significant lifecycle changes and major workflow decisions instead of showing every minor system operation.

## Aligning Diagrams with Agile User Stories
Another challenge was linking the diagrams to Agile artifacts from previous assignments. User stories are written from the perspective of stakeholders, while state diagrams focus on how system objects behave. To align them, I first mapped the diagrams to functional requirements and then linked those requirements to the related user stories and sprint tasks.

## Comparing State Diagrams and Activity Diagrams
State diagrams and activity diagrams model different aspects of system behavior. State diagrams focus on a single object and show how it changes over time in response to events. For example, an order moves through states such as Created, Confirmed, Shipped, and Delivered. Activity diagrams focus on workflows and show how multiple actors and system components participate in a process, such as checkout or payment processing.

Therefore, state diagrams are better for understanding object lifecycles, while activity diagrams are better for understanding process flow and responsibilities.
